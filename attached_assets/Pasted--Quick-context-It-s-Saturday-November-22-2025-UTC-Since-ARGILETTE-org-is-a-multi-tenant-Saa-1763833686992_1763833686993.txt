#### Core design principles for multi‑tenant AI agents
- **Strict tenant isolation by default**: Scope every retrieve, write, and tool-call by tenant_id; prevent cross-tenant data access at the app, data, and model layers. This is the recommended approach in cloud guidance for AI/ML multitenant solutions [[1]](https://learn.microsoft.com/en-us/azure/architecture/guide/multitenant/approaches/ai-ml). If you use managed agent services (e.g., Amazon Bedrock Agents), there are established patterns to enforce tenant isolation end-to-end [[2]](https://aws.amazon.com/blogs/machine-learning/implementing-tenant-isolation-using-agents-for-amazon-bedrock-in-a-multi-tenant-environment/).  
- **Per-tenant RAG + re-ranking**: Keep each tenant’s knowledge base and embeddings in its own namespace/index, and use a re-ranker to boost answer quality when multiple docs match—this is a standard step in robust RAG designs [[3]](https://aws.amazon.com/blogs/machine-learning/build-a-multi-tenant-generative-ai-environment-for-your-enterprise-on-aws/).  
- **Tenant-aware orchestration**: Route every agent/tool call with an explicit Tenant-ID and Authorization so the right tools, prompts, and indices are used per tenant; you can enforce this via a gateway/MCP server pattern that injects these headers and routes securely [[4]](https://www.novumlogic.com/blog/build-a-dynamic-multi-tenant-saas-platform-with-ai-agents-and-a-custom-mcp-server-client).  
- **Adaptive learning loops per tenant**: Capture outcomes (opens, replies, meetings, wins) and feed them back to update prompts, playbooks, and next-best-actions—agents improve using closed-loop feedback over time [[5]](https://www.aalpha.net/blog/how-to-integrate-ai-agents-into-a-saas-platform/).  
- **Multi-tenant agent posture**: The goal is one deployment serving many customers with strict data and operation isolation across tenants, which is the defining property of production-grade multi-tenant AI agents [[6]](https://ingenimax.ai/blog/building-multi-tenant-ai-agent).  
- **Workload isolation and scaling**: Use predictive autoscaling/queues to avoid tenant “noisy neighbor” issues; research proposes AI-orchestrated workload isolation with predictive analytics and RL to keep performance predictable under load [[7]](https://journalwjaets.com/sites/default/files/fulltext_pdf/WJAETS-2025-1310.pdf).

#### What “learning per tenant” looks like
- Per-tenant memory
  - Knowledge: product docs, FAQs, playbooks, prior proposals (embedded into a tenant namespace).
  - Interaction memory: conversation snippets, email threads, objections/resolutions.
  - Outcomes: meetings booked, win/loss reasons, top-performing posts/templates.
- Adaptive loop
  1) Log each action + outcome (e.g., reply intent, meeting, won/lost).
  2) Update per-tenant scoring models/rules and regenerate “best templates” and “next-best actions.”
  3) Refresh RAG snippets and summaries nightly; promote well-performing variants. This is the “adaptive learning loop” pattern for agent efficiency [[5]](https://www.aalpha.net/blog/how-to-integrate-ai-agents-into-a-saas-platform/).
- Safety boundary
  - No cross-tenant sharing unless you opt-in to an aggregated, anonymized global model; otherwise, keep learning purely tenant-local (also a standard multitenant AI posture [[1]](https://learn.microsoft.com/en-us/azure/architecture/guide/multitenant/approaches/ai-ml) [[6]](https://ingenimax.ai/blog/building-multi-tenant-ai-agent)).

```ts
// Node/Express example
app.use(async (req, res, next) => {
  const tenantId = req.header('Tenant-ID');
  const token = req.header('Authorization');
  if (!tenantId || !token) return res.status(401).send('Missing tenant or auth');
  const claims = await auth.verify(token, tenantId); // ensures token grants access to tenantId
  req.ctx = {
    tenantId,
    vectorNamespace: `vec_${tenantId}`,
    s3Prefix: `tenants/${tenantId}/`,
    modelProfile: await loadModelProfile(tenantId), // which model, temperature, system prompts
    rateLimiter: getLimiter(tenantId)
  };
  next();
});
```

#### Per-tenant RAG + re-ranking (skeleton)
```python
def answer_question(tenant_id, query, k=12):
    # 1) Retrieve from tenant's namespace
    cands = vectordb.search(namespace=tenant_id, query=query, top_k=k)
    # 2) Re-rank for relevance (e.g., cross-encoder)
    reranked = reranker.rank(query, cands)  # improves context quality [[1]]
    # 3) Compose prompt with tenant system prompt
    sys = ai_config.system_prompt(tenant_id)
    ctx = "\n\n".join([r.text for r in reranked[:5]])
    return llm.generate(system=sys, user=f"Question: {query}\n\nContext:\n{ctx}")
```
